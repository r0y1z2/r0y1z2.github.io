前言
随着互联网的发展，单机应用已经无法满足用户的需求，因此集群应用而生
本章将主要介绍高可用集群与负载均衡集群的实现工具的配置，包括LVS、HAProxy、Nginx和Keepalived 。
目标

  • 熟悉LVS的搭建和配置

0x01 集群

1.1 什么是集群

使用两台及两台以上的计算机完成一个任务的模式称为集群。在互联网发展迅猛的今天，单机的模式已经不再适用，集群已经成为当前最主要的业务构建模式

常见的集群类型包括：

  • LB集群
    • LB全称为Load Balance，LB集群按照不同算法将前端的访问转发给不同的后端计算节点，是所有的节点的负载达到相对平衡的状态
  • HA集群
    • HA全称为Highly Available，HA集群用来保障应用持久，不间断的提供服务，当应用程序出现故障，或者系统硬件、网络出现故障时，应用可以快速地从一个节点切换到另一个节点
  • HPC集群
    • HPC全称为High Performance Computing，HPC是一套软、硬件协同工作的系统，它是计算机科学的一个分支通过集群架构、并行算法和相关软件以并行计算或者分布式计算的方式实现单台计算机无法达到的运算速度(每秒万亿次以上)
  • 计算机集群（英语：computer cluster）是一组松散或紧密连接在一起工作的计算机。由于这些计算机协同工作，在许多方面它们可以被视为单个系统。与网格计算机不同，计算机集群将每个节点设置为执行相同的任务，由软件控制和调度
  • 集群的组件通常通过快速局域网相互连接，每个节点（用作服务器的计算机）运行自己的操作系统实例。在大多数情况下，所有节点使用相同的硬件和相同的操作系统
  • 部署集群通常是为了提高单台计算机的性能和可用性，而集群也通常比速度或可用性相当的单台计算机的成本效益要高

1.2 基于Linux的负载均衡集群软件

  • LVS

    • 全称为Linux Virtual Server，是一款开源的四层（传输层）负载均衡软件

  • Nginx

    • 支持四层和七层（应用层）负载均衡

  • HAProxy

    • 和Nginx一样，也可同时支持四层和七层（应用层）负载均衡

  • Nginx和HAProxy虽然支持四层负载均衡，但在大多数场景中，它们都会被选择为七层负载均衡

1.3 基于Linux的高可用集群软件

  • Keepalived
    • Keepalived是Linux中一个轻量级高可用集群部署软件，最初是为LVS设计的，后加入了VRRP功能，可以实现集群的心跳检测、虚拟IP漂移等功能
  • Heartbeat
    • Heartbeat的功能与keepalived功能类似，但是不控制虚拟IP的漂移，相对Keepalived更擅长对资源服务的控制，配置也比较复杂
  • Pacemaker
    • Pacemaker是一款在Linux中应用非常广泛的开源集群资源管理器，它本身不具备集群心跳检测机制，需依赖Corosync或Heartbeat，可同时适用于任何规模的集群
  • 在linux中常见的心跳机制使用的软甲有keepalived和heartbeat，openEuler二者都支持。Keepalived在安装、配置、使用和维护等方面上比heartbeat更容易，heartbeat相对来说要复杂，但是功能更强大，配套的工具也更加全面。找中小规模的集群中，使用keepalived，在大型集群中，需要使用heartbeat。

1.4 集群中的常见概念

1.4.1 VIP和地址漂移

  • VIP:虚拟IP。在高可用集群中，一般使用VIP向外提供服务，可避免集群中某台主机宕机后可能引起的业务访问IP发生变化
  • 地址漂移:在高可用集群中，VIP从某一台主机迁移到另外一台主机称为地址漂移
  • 案例：Server A和Server B组成一个集群，二者的真实地址分别是192.168.31.11和192.168.31.12，二者通过高可用集群软件配置了一个虚拟IP（VIP）为192.168.31.112，Client的ip地址为192.168.31.13，我们的目标无论是Server A还是Server B出现宕机，Client都可以ping通192.168.31.112

1.4.2 心跳和脑裂

  • 心跳:在高可用集群中，主机之间会相互监控彼此对方的状态，以决定哪台主机提供服务，这一机制称为心跳
  • 脑裂:如果节点之间的心跳通信中断或不正常，会导致彼此之间抢夺资源，这种现象称为脑裂
  • 一款好的高可用集群软件，可有效的避免脑裂现象

1.4.3 代理

  • 代理可分为正向代理和反向代理，一般用于负载均衡集群
  • 正向代理和反向代理是相对于被代理者来说的，正向代理被代理的是客户端，反向代理被代理的是服务端

0x02 常见LB集群部署工具

2.1 LVS简介

2.1.1 LVS在企业IT架构中的位置

  • VIP为虚拟IP，是LVS集群对外提供服务的IP地址
  • DIP为Director IP，用于连接业务集群，外部网络不能直接通信，业务请求需要经过VIP后转发给DIP
  • RIP为Real IP，是真实的业务主机（Real Server缩写为RS）IP地址，和DIP之间可达
  • LVS本身只能完成负载均衡的调度作用，容易形成单点故障，因此需要和Keepalive配合使用，形成LVS集群，才能适用于实际的业务场景，消除单点故障

2.1.2 LVS的组成及作用

  • LVS由两部分组成，分别是：
    • lpvs:ip virtual server，运行在内核当中，基于内核态的netfilter框架实现
    • lpvsadm:运行在用户态，是LVS的管理工具，可以实现LVS的配置，例如VIP的增删查改等lpvsadm命令格式
  • lpvsadm命令格式

2.1.3 Ipvsadm常用操作

  • 清除所有规则
    • lpvsadm -C
  • 保存规则到配置文件
    • `lpvsadm -Sn > /etc/sysconfig/ipvsadm
  • 从配置文件加载规则
    • lpvsadm -R < /etc/sysconfig/ipvsadm
  • 查看当前规则
    • lpvsadm -Ln

2.1.4 LVS的三种工作模式和十二种调度算法

  • LVS的三种工作模式
    • LVS/NAT
    • LVS/DR
    • LVS/TUN
  • LVS的十二种调度算法
    • RR
    • WRR
    • SH
    • DH
    • LC
    • WLC
    • SED
    • NQ
    • LBLC
    • LBLCR
    • FO
    • OVF
  • NAT和DR模式比较常用，实验手册只进行这两种的配置
  • FO和OVF为4.15版本中新增的调度算法，暂时用的比较少，后续不再单独介绍

2.1.5 LVS的工作模式 - NAT

  • 假设User的IP地址为192.168.1.10，LVS的DIP地址为10.0.0.20，服务端口为80，RS1的地址为172.16.2.100，服务端口为8443
  1. User向LVS发起请求，源地址为192.168.1.10，目的地址为LVS的VIP目的端口号为80
  2. LVS收到请求后，进行调度，并重写数据包，源地址不变，把目的地址修改为172.16.2.100，目的端口号修改为8443
  3. RS收到请求后进行处理，并在响应报文中把源地址修改为自己的地址172.16.2.100，目的地址修改为User的地址192.168.1.10，源端口设置为8443
  4. LVS收到响应报文后，再次对报文进行修改，源地址修改为自己的VIP，源端口为80，目标地址为User的192.168.1.10，目的端口为User发请求是的随机端口
  5. LVS将修改后的报文响应给User端
  • 图片来源：http://www.linuxvirtualserver.org/VS-NAT.html
  • LVS-NAT模式的本质是将请求报文中的目标地址和目标端口按照配置和算法，转发给不同业务主机的地址和端口
  • NAT模式下，所有的报文都经过LVS服务器，因此会导致其成为整个环节的瓶颈，因此该模式适用于后端服务器较少的场景中

2.1.6 LVS/NAT模式配置案例

  • 开启LVS的地址转发功能
  • 增加一个集群
  • 添加后端RS 检查配置后状态 在命令ipvsadm -A -t 192.168.1.10:80 -s rr中，-A表示增加一个集群，-t表示使
    用TCP协议，-s rr表示使用的调度算法为rr

2.1.7 LVS的工作模式

2.1.7.1 DR

  • 在DR模式中，所有的RS需要配置两个地址:RIP和VIP

  • 在DR模式中，LVS会通过arp获取到所有RS的IP地址和对应的MAC地址，因此LVS和RS需处于同一二层网络中

  • 当User发送请求到LVS后，LVS保持源目IP地址和端口号不变同时将源MAC地址换成自己的MAC地址，目的MAC地址换成调度到的RS MAC进行转发

  • RS在回响应报文时，源地址是VIP地址，源端口号为提供服务的端口号，源MAC地址为自己的MAC地址，目的地址为客户端地址，目的端口号为客户端发起连接请求时的随机端口，目的MAC地址为网关的MAC地址

  • DR是LVS的默认模式，它的本质是重新封装请求报文，将源MAC地址替换为DIP所在接口的MAC地址，目标MAC地址设置为RS的RIP所在接口的MAC地址，源目IP和端口都保持不变。在这个模式下，仅有请求报文经过LVS，响应报文则不经过LVS，这样就可以减轻LVS的负荷，提高整个业务流程的性能

  • RS上配置的VIP需和LVS的VIP保持一致，为了防止地址冲突，需要在RS上关闭VIP所承载网卡的arp广播功能和arp应答功能

  • 所有的VIP一般都配置在回环口上

  • 在DR模式中，LVS的DIP和RS的RIP需在同一网段。RIP和VIP可以不在同一网段

  • DR模式中，在LVS上不支持修改端口号

  • DR模式中，由于要求DIP和RIP在同一网段，因此无法进行大规模或远距离部署

LVS/DR模式配置案例

  • 在LVS和RS上配置VIP
  • 在RS上配置VIP所在网卡arp对应的功能
  • 在LVS和RS上配置VIP
  • 添加集群 echo 1 > /proc/sys/net/ipv4/conf/……四条命令仅修改了临时配置文件，重启后，配置会消失，如果要永久生效，需将配置写入配置文件中
2.1.7.2 TUN
  • TUN模式和DR模式类似，区别在于允许DIP和RIP在不同的网段，因此请求报文到达LVS后，LVS会在其基础上添加一个隧道头 ，里面包含了源地址为DIP、目的地址为RIP的信息
  • 经过隧道后，数据包到达对应的RS后，它会对隧道头进行解封装，露出其中的报文头及报文内容，然后进行处理和响应
  • 响应报文的源地址为VIP，目的地址为客户端地址
  • RS在使用VIP进行响应，因此，和DR一样，RS上也需要配置VIP
  • TUN模式的优势LVS和RS之间可以跨网段，甚至可以跨广域网

2.1.8 LVS的调度算法 - 静态算法

  • LVS的静态调度算法一共有四种，分别是：
    • RR：轮询算法
      • 调度器将收到的请求按顺序轮流分配到集群中的RS。均匀地分配请求，不考虑服务器连接数和负载情况
    • WRR：加权轮询算法
      • 根据RS的处理能力给予不同的权重，为性能好的服务器分配更多的请求，性能一般的分配较少的请求
    • SH：源地址哈希算法
      • 根据请求的源IP地址作为哈希键从静态分配的哈希表中找出对应的服务器。同一个IP地址的请求始终发往第一次调度的RS，可实现会话绑定
    • DH：目的地址哈希算法
      • 根据请求的目标IP地址作为哈希键从静态分配的哈希表中找出对应的服务器。发往同一个目标地址的请求始终转发到第一次调度的RS，可用于实现正向代理缓存场景中的负载均衡
    • LVS已经集成在Linux内核中，使用命令：grep -i -C 10 ipvs /boot/config-5.10.0-60.18.0.50.oe2203.x86_64可查看到关于LVS支持的算法

2.1.9 LVS的调度算法 - 动态算法

  • LVS的动态调度算法一共有八种，分别是：
    • LC：最少连接算法
      • 动态地网络请求调度到连接数最少的RS上。当集群中的服务器性能相近时，这种算法可以进行较好的均衡负载。适用于长链接应用。其负载计算公式为：overhead=活动连接数*256+非活动连接数
    • WLC：加权最少连接算法
      • 相对于LC，当集群中服务器性能有差异时，建议使用本算法，可以使性能较好的服务器承接较大比例的活动连接。此算法是LVS的默认调度算法，负载公式为:overhead = (活动连接数*256+非活动连接数)/权重值
    • SED：最短期望延迟算法
      • 当使用WLC时，如果RS的连接数和权重比例正好一样时，下个请求将可能分配给任意一个RS，而不是性能最好的RS，SED可避免这种情况，保证了高权重优先
    • NQ:最少队列算法
      • 使用该算法时，第一轮请求会均匀的分配给所有RS，后续再使用SED算法，这样可以避免某些服务器可能用于得不到处理请求的机会
    • LVS已经集成在Linux内核中，使用命令：grep -i -C 10 ipvs /boot/config-5.10.0-60.18.0.50.oe2203.x86_64可查看到关于LVS支持的算法

2.2 Nginx简介

2.2.1 Nginx的负载均衡功能

  • Nginx可以提供四层和七层的负载均衡
  • Nginx提供了多种负载均衡算法，如轮询、权重等
  • 轮询算法下，Nginx会轮流将请求发送给不同的服务器，使每个服务器都均等的提供服务
  • 权重算法下，Nginx会按照配置的权重比例，将请求发送给服务器。性能较好的服务器将提供更多的服务，性能较好的服务器将提供较少的服务

2.2.3 Nginx支持的算法

  • RR
    • 轮询，Nginx的默认算法
  • LC
    • 最少连接调度
  • IP Hash
    • 基于源地址的额哈希值选择合适服务器
  • Generic Hash
    • 基于自定义的key的哈希值来实现调度，key支持文本、变量或者文本+变量的方式指定
  • Random
    • 随机指定，该算法会从后端服务器中随机指定一个来响应客户端请求。在该算法下，后面可加选项“two”，表示随机挑选两台服务器出来，然后根据指定的算法来响应请求，如果不指定算法则参考权重来响应
  • Nginx还支持Least time算法，仅在Nginx plus（商业版）支持，它在最少连接数算法的基础上增加了响应时间，因此使用时需要加上header或者last_byte

2.2.4 Nginx的负载均衡的其他指令参数

  • Nginx的upstream模块还提供了一些其他指令参数，对负载均衡进行配置
  • max_conns num:设置被负载的服务器的最大连接数
  • down:常用于IP Hash算法，表示对应的服务器状态为down，Nginx将不会把请求转发到对应服务器
  • backup:表示对应的服务器为备份状态，当其他服务器全部出现故障后，Nginx才会将请求转发到对应的服务器
  • max_fails和fail timeout:用于健康检查，其中max_fails表示Nginx对后端服务器请求失败的次数，默认值为1，如果1次失败后，Nginx就会在failtimeout所设定的时间内，不会将请求转至该服务器。在failtimeout时间后，nginx会尝试的把新的请求转发至该服务器，如果仍然没有恢复正常，Nginx就继续在failtimeout所设定的时间内，不会将请求转至该服务器
  • slow_start:用于健康检查中，当集群中有服务器在等待fail_timeout后，Nginx检测到该服务器恢复后，仍不会立刻转发请求到该服务器上，而是等待slowstart指定的时间后，才会再转发请求到该服务器上
  • Slow_start在Nginx plus（商业版）中可用

2.2.5 Nginx的反向代理功能

  • Nginx反向代理的工作流程:
  1. 客户端向Nginx发送请求
  2. Nginx在收到客户端发送的请求后，将请求转发到后端服务器
  3. 后端服务器将客户端请求的资源回复给Nginx
  4. Nginx将资源返回给客户端
  • 在反向代理的工作流程中，客户端向代理请求资源时使用的端口，可以和服务器提供服务的端口不一致
  • Nginx支持对http、https、tcp、websocket、GRPC、pop/imap和rtmp等协议进行代理
  • 例如：客户端使用80端口向代理发送请求，服务器使用8080端口向反向代理提供资源

2.2.6 Nginx的反向代理配置

  • Nginx反向代理配置位于location模块中
  • 使用proxy_pass实现反向代理的语法为：proxy_pass URL;
  • 当proxy_pass指定了一个具体url后，location中的路径会被取代
  • 当proxy_pass未指定具体url后，代理会保留客户端发来的请求格式转发给服务器，或者在处理更改URI时传递完整的标准化请求URI
  • 在配置反向代理时，location中路径和proxy_pass中url包不包含“/”会对后面转发有很大影响，假设要访问的地址为http://192.168.1.123/test/index.html，不同写法会造成不同的转发路径，具体如下：
    • location /test/ { proxy_pass http://192.168.1.123/ ;}，最终转发路径为：http://192.168.1.123/index.html
    • location /test/ { proxy_pass http://192.168.1.123 ;}，最终转发路径为：http://192.168.1.123/test/index.html
    • location /test { proxy_pass http://192.168.1.123/ ;}，最终转发路径为：http://192.168.1.123//index.html
    • location /test { proxy_pass http://192.168.1.123 ;}，最终转发路径为：http://192.168.1.123/test/index.html

2.3 HAProxy简介

2.3.1 HAProxy的负载均衡功能简介

  • 负载均衡是HAProxy基本功能之一
  • HAProxy支持四层和七层的负载均衡HAProxy基于事件驱动的链接处理模式和单进程处理模式，使其性能更优
  • HAProxy支持ACL功能
  • HAProxy支持图形化监控
  • HAProxy支持不少于10种的调度算法

2.3.2 HAProxy支持的通用调度算法简介

  • RR
    • 轮询调度，默认算法
  • Source
    • 基于源地址hash，取值后如果源地址相同，则转发至相同服务器
  • URI
    • 基于用户请求的uri进行hash计算，取值后将请求转发至指定服务器
  • HDR
    • 针对用户http头部请求中的指定信息进行hash计算，然后根据权重将请求转发至对应后端服务器，如果指定的值无效，则会使用默认的轮询算法进行调度
  • First
    • 根据配置文件中服务器的列表自上而下调度，当第一台服务器连接数到达上限后，新的请i去会被转发至下一台服务器，以此类推
  • Leastconn
    • 最少动态连接算法，优先将新的请求转发至活动连接数最少服务器上

2.3.3 HAProxy调度算法特征

  • HAProxy调度算法都支持手动指定不同的权重，但是无法实时修改，需重启HAProxy才能生效
  • RR、leastconn和hash算法支持动态权重调整，无需重启HAProxy
  • 动态权重调整的算法也同时支持慢启动(slow-start)Hash算法取值的对象支持多种元素，例如客户端地址、URL组件、请求字符元素、头文件值、POST参数和RDP cookie
  • 支持强一致性hash算法
  • 支持多种指标，可构建非常高级的负载平衡策略

2.3.4 HAProxy配置文件组成

  • HAProxy配置文件（/etc/haproxy）由以下部分组成：
    • global
      • 用于定义全局参数，属于进程级的配置，通常和操作系统配置有关
    • default
      • 用于配置默认参数，这些参数可以被用到fronted，backend，listen组件
    • frontend
      • 用于定义前端服务的配置。在frontend中配置ACL规则，可将请求定向到相关的backend
    • backend
      • 用来定义前端服务集群的配置，真实服务器，一个backend对应一个或者多个实体服务器
    • listen
      • 常常用于状态页面监控，以及后端server检查，是frontend和backend的组合体

2.3.5.1 HAProxy ACL规则配置

HAProxy ACL定义语法格式为：acl <aclname> <criterion> [flags] [operator] [<value>]

  • <aclname>为自定义的规则名，可使用字母、数字和特殊符号，字母区分大小写
  • <criterion>为匹配条件，HAProxy支持域名、url、源地址、目标地址等多种匹配条件
  • [flags]为条件标标志位
  • [operator]为具体操作符
  • [<value>]为操作对象类型

匹配条件说明：

  • hdr(host)：匹配域名全称，括号里面可以是http请求头里面其他字段，如User-Agent，

  • hdr_reg(host)：正则匹配域名

  • hdr_dom(host)：匹配域名全名

  • hdr_beg(host)：匹配域名前缀

  • hdr_end(host)：匹配域名后缀

  • hdr_len(host)：匹配域名长度

  • hdr_sub(host)：匹配域名子串

  • hdr_dir：匹配域名dir

  • url_sub：匹配url子串，同hrd一样有8种匹配模式

  • path_end：匹配path后缀，同hdr一样有8种匹配模式，path是指请求的URL路径，该路径从第一个斜杠开始，并在问号之前结束（无主机部分），默认会被调度到同一后端主机上，完整的url中：
    <scheme>://<user>:<password>@<host>:<port>/<path>; ?#，path指/;`

  • base_beg：匹配base前缀，同hrd一样有8种匹配模式，base是指第一个主机头和请求的路径部分的连接，该请求从第一个斜杠开始，并在问号之前结束，对虚拟主机有用，完整的url中：<scheme>://<user>:<password>@<host>:<port>/<path>;<params>?<query>#<frag>，base指的是<host>:<port>/<path>;<params>

    • src ：匹配源ip
    • src_port：匹配源端口
    • Dst：匹配目标ip
    • dst_port：匹配目标端口
    • Method：匹配http请求方式：GET\POST…

  • Flag说明

    • -i：不区分大小写
    • -m：使用指定的pattern匹配方法
    • -n：不做DNS解析
    • -u：禁止acl重名，否则多个同名ACL匹配
    • –：强制flag结束. 当字符串和某个flag相似时使用

Operator说明

  • 整数类：
    • eq：等于；ge：大于等于；gt：大于；le：小于等于；lt：小于
  • 字符串：
    • exact match (-m str)：完全匹配
    • substring match (-m sub)：匹配子串，若其中任何一个被发现，ACL将匹配
    • prefix match (-m beg)：匹配前缀，若其中任何一个被发现，ACL将匹
    • 配
    • suffix match (-m end)：匹配后缀，若其中任何一个被发现，ACL将匹配
    • subdir match (-m dir)：匹配用斜线分隔（“/”）的字符串，若其中任何一个匹配，ACL将匹配
    • domain match (-m dom)：匹配用点（“.”）分隔的字符串，若其中任何一个匹配，ACL将匹配

Value说明：

  • boolean：布尔值false、true
  • integer or integer range：整数或整数范围，比如用于匹配端口范围，1024～32768 ；IP地址或IP范围， 192.168.0.1 ，192.168.0.1/24；长度范围
  • String：域名、path、url…的全名、前缀、后缀子串…
  • regular expression：正则表达式
  • hex block：16进制
2.3.5.2 HAProxy ACL规则调用
  • 在ACL规则调用时，遵循以下逻辑关系:
    • 与:隐式使用，默认
    • 或:使用“or”或“|”表示
    • 非:使用“!”表示
  • 规则调用语句说明:
    • use_backend <backend>[{if|unless}<condition>]，表示满足条件时接收请求
    • block {if|unless}<condition>，表示满足条件时拒绝请求
    • http-request fallow | auth [realm <realm>]| redirect <rule> | deny [deny _status <status>]} [{if| unless} <condition>]], 用于七层调度，allow表示接受该请求，auth表示提示输入用户认证信息，指定realm时将给出提示信息，redirect表示重定向功能，deny表示拒绝该http请求
    • tcp-request connection {accept|reject}[{if|unless} <condition>]，用于四层调度，表示满足条件时接收或拒绝请求
    • tcp-response connection < acceptlreiectlclose>fiflunless} <condition>]。用于四层调度，表示满足条件时接收或拒绝响应报文,close表示满足条件时立即关闭和服务器的连接

3.3 VRRP功能简介

  • VRRP（VirtualRouterRedundancyProtocol）：虚拟路由冗余协议，解决静态网关单点故障风险，它能够保证当个别节点宕机时，整个网络可以不间断地运行

  • 图片来源： https://support.huawei.com/enterprise/zh/doc/EDOC1100209553/

3.3.1 VRRP工作原理

  • VRRP协议中定义了三种状态机：初始状态（Initialize）、活动状态（Master）、备份状态（Backup）。其中，只有处于Master状态的设备才可以转发那些发送到虚拟IP地址的报文

3.3.2 VRRP选举机制

  • 图片来源： https://support.huawei.com/enterprise/zh/doc/EDOC1100209553/
  • VRRP详细工作过程
    • VRRP备份组中的设备根据优先级选举出Master。Master设备通过发送免费ARP报文，将虚拟MAC地址通知给与它连接的设备或者主机，从而承担报文转发任务。
    • Master设备周期性向备份组内所有Backup设备发送VRRP通告报文，通告其配置信息（优先级等）和工作状况。
    • 如果Master设备出现故障，VRRP备份组中的Backup设备将根据优先级重新选举新的Master。
    • VRRP备份组状态切换时，Master设备由一台设备切换为另外一台设备，新的Master设备会立即发送携带虚拟路由器的虚拟MAC地址和虚拟IP地址信息的免费ARP报文，刷新与它连接的设备或者主机的MAC表项，从而把用户流量引到新的Master设备上来，整个过程对用户完全透明。
    • 原Master设备故障恢复时，若该设备为IP地址拥有者（优先级为255），将直接切换至Master状态。若该设备优先级小于255，将首先切换至Backup状态，且其优先级恢复为故障前配置的优先级。
    • Backup设备的优先级高于Master设备时，由Backup设备的工作方式（抢占方式和非抢占方式）决定是否重新选举Master。
